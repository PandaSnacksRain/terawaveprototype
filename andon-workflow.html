Workflow Specification

1. What event starts the workflow?

Operator pulls andon cord/button at their station when they encounter an issue they cannot resolve themselves.

Trigger: Physical button press or UI "Pull Andon" action

Input data: Station ID, WO number, operator ID, timestamp, current op step

2. What business object is moving through the process?

Primary: Andon Ticket

Attributes: Ticket ID, Station, WO#, Serial#, Issue Type, Duration, Assigned Responder, NC Number, Resolution

Linked objects: Work Order, Unit/Serial, NC Record, SOP, Responder

3. What states can it be in?

OPEN → Andon pulled, awaiting triage

TRIAGED → Issue classified, awaiting assignment

ASSIGNED → Responder assigned, awaiting action

IN PROGRESS → Repair/resolution underway

RESOLVED → Issue fixed, pending close-out

CLOSED → Documentation complete, metrics updated

TERMINATED → Invalid/cancelled andon

4. Who or what acts at each state?

OPEN: System notifies supervisor

TRIAGED: Supervisor classifies issue type

ASSIGNED: System auto-assigns responder by cert + availability

IN PROGRESS: Responder executes repair per SOP

RESOLVED: Responder marks complete

CLOSED: System generates NC, updates WO, feeds metrics

5. What rules determine branching?

validate_andon: Is WO active and station valid? → false: terminate

confidence_gate: AI confidence >85%? → low: require human triage

resolution_gate: Based on issue severity + estimated repair time:

• <15 min fixable → In-line repair

• Needs part removal → Hospital bay

• Beyond tech scope → Escalate to engineering

complete_gate: Repair successful? → false: retry repair

6. Where does an agent add value?

ai_suggest_ref: Predicts issue type from station history, part data, and symptom patterns. Shows confidence % so supervisor knows when to trust vs. override.

filter_responders_ref: Ranks available responders by certification match, proximity, and current workload.

auto_assign_ref: Suggests best responder with ETA.

create_nc_ref: Auto-generates NC record with pre-filled fields from andon context.

7. Where does a human need to decide?

human_triage_ref: Supervisor confirms or overrides AI-suggested issue type when confidence is low or edge case.

resolution_gate_ref: Supervisor chooses resolution path (in-line / hospital / escalate) based on judgment of severity and shop floor context.

execute_sop_ref: Responder performs physical repair, makes real-time decisions during execution.

complete_gate_ref: Responder judges if repair is truly complete and unit is good.

8. What does the UI need to surface at each handoff?

OPEN → TRIAGED: Andon alert card with station, duration timer, WO context, AI prediction with confidence bar, issue type buttons

TRIAGED → ASSIGNED: Responder list filtered by cert, availability status, ETA, one-click assign

ASSIGNED → IN PROGRESS: Resolution path cards (in-line/hospital/escalate) with time estimates, SOP preview

IN PROGRESS → RESOLVED: Active repair view with SOP steps, timer, parts used, "Mark Complete" button

RESOLVED → CLOSED: Resolution form with root cause dropdown, notes field, auto-populated NC preview
